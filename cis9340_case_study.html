Project Overview

As part of the Principles of Database Management Systems course, we were tasked with creating a fully functional relational database for a real‑world domain. Our team chose to design a Hospital Management System to support patient intake, appointment scheduling, billing, and record management.

The project requirements included:

  • Conducting a system analysis and defining use cases and functional requirements.
  • Developing an Entity–Relationship (ER) diagram that accurately models the domain.
  • Normalizing the schema to 3rd Normal Form (3NF) to eliminate redundancy.
  • Implementing the database in MySQL with tables, keys, and constraints.
  • Populating tables with realistic sample data and writing SQL queries to answer business questions.
  • Creating stored procedures, triggers and views to automate logic and encapsulate business rules.

Designing the Schema

We began by interviewing stakeholders (imaginary in this context) to identify the key entities: Patients, Appointments, Doctors, Departments, and Bills. Relationships such as one‑to‑many (a patient can have many appointments) and many‑to‑one (many appointments belong to one doctor) were mapped out.

Our ER diagram captured these relationships and highlighted important attributes like patient demographics, appointment times, and billing codes. After normalizing the schema to eliminate duplication, we implemented the following tables (simplified here):

CREATE TABLE Patients (
  patient_id INT PRIMARY KEY,
  first_name VARCHAR(50),
  last_name VARCHAR(50),
  date_of_birth DATE,
  gender CHAR(1),
  phone VARCHAR(20),
  address VARCHAR(255)
);

CREATE TABLE Doctors (
  doctor_id INT PRIMARY KEY,
  name VARCHAR(100),
  specialty VARCHAR(100),
  department_id INT,
  FOREIGN KEY (department_id) REFERENCES Departments(department_id)
);

CREATE TABLE Appointments (
  appointment_id INT PRIMARY KEY,
  patient_id INT,
  doctor_id INT,
  appointment_datetime DATETIME,
  notes TEXT,
  FOREIGN KEY (patient_id) REFERENCES Patients(patient_id),
  FOREIGN KEY (doctor_id) REFERENCES Doctors(doctor_id)
);

Querying & Integrity

Once the schema was built and populated with sample data, we wrote SQL queries to support common operational and analytical tasks. Examples include listing all upcoming appointments for a doctor, counting patients seen by department, and calculating revenue per month.

-- Find all appointments for Dr. Smith next week
SELECT a.appointment_id, p.first_name, p.last_name, a.appointment_datetime
FROM Appointments a
JOIN Patients p ON a.patient_id = p.patient_id
JOIN Doctors d ON a.doctor_id = d.doctor_id
WHERE d.name = 'Dr. Smith'
  AND a.appointment_datetime BETWEEN '2025-11-01' AND '2025-11-07'
ORDER BY a.appointment_datetime;

-- Total revenue per department (simplified)
SELECT d.department_name, SUM(b.amount) AS total_revenue
FROM Bills b
JOIN Doctors doc ON b.doctor_id = doc.doctor_id
JOIN Departments d ON doc.department_id = d.department_id
GROUP BY d.department_name;

Triggers and stored procedures were added to enforce business rules; for example, a trigger prevented inserting an appointment that overlapped with another appointment for the same doctor. Views were created to simplify reporting.

Reflection

This project underscored the importance of thoughtful data modeling and normalization. It also highlighted how SQL can be used not only for data retrieval but also for enforcing integrity and implementing business logic. By connecting this project to our work in programming for analytics, we realized how database design decisions impact data retrieval efficiency and subsequent analysis.
